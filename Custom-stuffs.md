# Custom Model and Dataset Integration Guide

## Overview

This benchmarking framework evaluates OCR, OMR, and layout detection models on historical document
analysis tasks. The system supports multiple frameworks (Kraken, Calamari, TrOCR, YOLO, DETR,
Faster-RCNN) through a standardized interface that allows seamless integration of custom models.
It uses COCO-based annotations for representing the ground-truth data and generates the necessary
splits in a reproducible way.

## Add a Custom Dataset

### Dataset Structure Requirements

```
your_dataset/
├── images/                    # All image files (any format)
│   ├── page_001.png
│   ├── page_002.jpg
│   └── ...
└── annotations/               # COCO-format annotation files
    ├── gt.json               # MASTER annotation file (required)
    └── processed_splits/     # Auto-generated by framework
```

### COCO Annotation Format (Mandatory)

Create `gt.json` with this exact structure:

```json
{
  "images": [
    {
      "id": 1,                    // Unique integer
      "width": 1000,              // Image dimensions in pixels
      "height": 1500, 
      "file_name": "images/page_001.png"  // Relative path from dataset root
    }
  ],
  "annotations": [
    {
      "id": 1,                    // Unique integer  
      "image_id": 1,              // MUST match images[].id
      "category_id": 6,           // MUST use framework category IDs
      "bbox": [x, y, width, height],  // [x, y, width, height] in pixels
      "description": "text content",  // REQUIRED for OCR/OMR tasks
      "iscrowd": 0,
      "area": width * height      // Optional but recommended
    }
  ],
  "categories": [                 // CRITICAL: Use exact IDs and names
    {
      "id": 4,
      "name": "text",             // MUST be "text"
      "supercategory": "layout"   // MUST be "layout"
    },
    {
      "id": 5, 
      "name": "staff",            // MUST be "staff"
      "supercategory": "layout"   // MUST be "layout"
    },
    {
      "id": 6,
      "name": "line",             // MUST be "line" 
      "supercategory": "layout"   // MUST be "layout"
    },
    {
      "id": 7,
      "name": "musicText",        // MUST be "musicText"
      "supercategory": "layout"   // MUST be "layout"
    }
  ]
}
```

### Category System (Fixed Mappings)

The framework uses **hardcoded category IDs** for automatic task detection:

#### ✅ Processed Categories

| Category ID | Name | Task | Description |
|-------------|------|------|-------------|
| 4 | `text` | Layout | Text regions |
| 5 | `staff` | OMR/Layout | Music staff lines |
| 6 | `line` | OCR/Layout | Text lines |
| 7 | `musicText` | OMR/Layout | Music notation |

#### ❌ Ignored Categories

- Any category where `supercategory != "layout"`
- Any category with ID not in [4, 5, 6, 7]
- Categories with different names than specified above

#### Task Auto-Detection

- **OCR**: Processes annotations with `category_id: 6` (text lines)
- **OMR**: Processes annotations with `category_id: 5` (music staff)
- **OCMR**: Processes annotations with `category_id: 5` OR `6` (both)
- **Layout**: Processes all categories `[4, 5, 6, 7]`

### Adding Custom Layout Categories

You can extend the layout task with custom categories:

```json
{
  "categories": [
    // Framework categories (REQUIRED)
    {"id": 4, "name": "text", "supercategory": "layout"},
    {"id": 5, "name": "staff", "supercategory": "layout"},
    {"id": 6, "name": "line", "supercategory": "layout"},
    {"id": 7, "name": "musicText", "supercategory": "layout"},
    
    // Custom categories (OPTIONAL)
    {"id": 8, "name": "illustration", "supercategory": "layout"},
    {"id": 9, "name": "margin_note", "supercategory": "layout"},
    {"id": 10, "name": "initial", "supercategory": "layout"}
  ]
}
```

**Then modify `ann_handler.py`** to include custom IDs:

```python
## In create_all_pagexml_files() function:
create_xml_for_task("layout", [4, 5, 6, 7, 8, 9, 10], pagexml_layout_dir)
```

### Automatic Processing Pipeline

The framework automatically:

1. **Detects dataset type** from path ("editorial" vs diplomatic)
2. **Filters annotations** by category ID for each task
3. **Creates splits** (5-fold cross-validation + sequential splits)
4. **Generates PageXML** for all tasks
5. **Handles empty data** (creates dummy entries when needed)

### Important Constraints

1. **Fixed Category System**: You MUST use the specified category IDs and names
2. **Relative Image Paths**: `file_name` must be relative to dataset root
3. **Text Content Required**: OCR/OMR annotations must include `description` field
4. **Bounding Box Format**: `[x, y, width, height]` in pixels
5. **Unique IDs**: All image and annotation IDs must be unique integers

### Troubleshooting

#### Common Issues

- **Missing descriptions**: OCR/OMR tasks will skip annotations without text
- **Wrong category IDs**: Annotations with invalid category IDs will be ignored
- **Path errors**: Ensure `file_name` paths are correct relative to dataset root
- **ID conflicts**: All IDs must be unique across images and annotations
